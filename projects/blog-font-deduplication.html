Font RAM Optimization in Unity

Stripping duplicate TextMesh Pro font memory on a standalone VR build

Related Unity thread

Issue Overview

TextMesh Pro (TMP) font assets in Unity have a requirement, that any font referenced via <font="Name"> rich-text tag or assigned as a TMP Settings default must live inside a Resources folder. At the same time, Addressables build pipeline packs anything reachable from a scene, prefab, or preloaded asset list into its own bundles. Any font asset that satisfies both systems at once eventually gets pulled into memory twice - once by the Resources loader at startup, once by the addressable bundle.

It may not seem like a huge issue, but considering Android VR hardware specifics, memory is one of the largest concerns, especially when it comes to older platforms like Quest 2. This, combined with Chinese/Japanese/Korean localization and different font width styles, project can easily end up with extra 150 MB of unused memory in RAM.

3-step solution

Build action: Font Stripping

A custom pre-build action that walks every prefab scene, material and removes font and it's material references and records where each one was.

Build action: Validation

A custom post-build action that fails the build step if any forbidden reference survives in Resources, scenes, preloaded assets, or always-included shaders.

Runtime: Font Loading

A loader that restores the stripped references from Addressables at runtime, after the build-only copy is already excluded.

Measured Impact

Tracking Down Every Reference

The hard part was never writing code that strips fonts in Resources. There, logic is straightforward, if you know how to work with looping over prefabs and active scenes, and have basic knowledge of Scriptable objects. The difficult part was finding every place, or category, where font references hide. TextMesh Pro spreads font and material references across more entities than inspector suggests. Hence, in the core of this blogpost, I presented a list, that actually mattered for this project:

Where Font References Actually Live

  • TMP_Text components - the font asset field itself, m_sharedMaterial and m_fontSharedMaterials fields
  • Custom text components wrapping a font set collection asset. Resolved through a separate indirection layer instead of TMP's own font field
  • TMP_SubMeshUI / TMP_SubMesh child objects, which TMP creates automatically for multi-atlas text and which carry their own independent font and material slots
  • TMP Settings default font asset and fallback font list - global, singleton-style references that pull a font into every build regardless of whether any scene uses it
  • Per-font fallback tables (m_FallbackFontAssetTable), where one font asset references another, creating a chain that the build pipeline treats as a hard dependency
  • Always Included Shaders in Graphics Settings and Preloaded Assets in Player Settings - neither shows up by browsing a scene, but both are root objects the build pipeline keeps alive unconditionally
  • Nested prefab variants, where the base prefab's font reference lives on the base asset and a variant's stripped override doesn't propagate backward - meaning the base has to be re-checked independently, not just walked transitively from the variant

But, list is NOT set in stone

Font references may live in other places, its very individual to the project. Hence, the validator exists. Every one of those surfaces can silently regress. To give an example "A teammate adds a fallback font, an artist drags a material back onto a header prefab to preview it" BOOM 💥 and the duplication is back. Quietly, with no error until someone runs a memory capture on-device weeks, or even worse, months later. Catching it at build time, with the exact dependency chain printed in the logs back to the developers is as important as the stripping logic itself.

The Other Half: Material Atlases

If you look at Memory Profiler picture carefully, you will notice memory improvements in Texture2DArray block. Stripping the font assets solved half the duplication. The other half was related batched UI material that samples six font atlases. 2 language types * 3 weights. Those were packed as individual textures on one shared material, which was used across the whole UI to keep draw calls low. The naive fix was a Texture2DArray built at import time, copying every atlas into one array texture. That works, but it at the cost of every atlas being copied: one inside each TMP_FontAsset, and one duplicated into the array.

To fix the issue I ensured the material's atlas slots are populated through OnGUI-stage code for batched UI material that reads each atlas directly off the live TMP_FontAsset objects and assigns them straight onto the material's texture properties. No array, no copy, the material points at the same atlas textures the font assets already own in memory. That way, the font assets stay the single place and batched material just has references to them.
